# OpenBench

A personal workspace for building projects, storing documents, and running agentic AI workflows.

## Purpose

- **Projects**: Develop software projects across various domains.
- **Documents**: Store and manage documents and reference materials.
- **AI/Agentic Workflows**: Use OpenCode for LLM-driven research, chat, coding, and agent-based automation.
- **Memory System**: Persistent agent memory for context retention across sessions.
- **Custom Agent Ecosystem**: Skills, commands, and plugins for extending agent capabilities.

### Working in `/work`

Most coding work happens inside `work/` sub-repos. Each directory there is an independently versioned git repo. Before working on a repo:

1. Check `memory/work-repos.md` for stack, conventions, and AGENTS.md link.
2. Read the repo's own `AGENTS.md` for project-specific commands, architecture, and deploy notes.
3. The repo may have its own `memory/` directory with persistent agent context.

Convention: `work/<repo>/` (shorter names preferred). Everything except `work/README.md` is gitignored at the subdirectory level so repos stay independently versioned.

### Search tools

Use these over `bash` + `grep` when possible — they're faster and burn fewer tokens:

- **Semble MCP** — Semantic code search across `work/` repos. Use `semble_search` (or the Semble MCP tool) for natural-language code queries ("find authentication logic", "where is the payment flow?"). Prefer over `grep` for conceptual searches; use `grep` only for exact string matching.
- **Session search** — Full-text search over past agent sessions. Use `bash` to run `python scripts/workspace/session-index.py search "<query>"`. Supports `--section`, `--project`, `--dates` filters. Use when the user asks about past discussions, decisions, or context. Skill: `session-search`.

## Scheduled jobs

OpenBench Cron runs automated agent tasks on a schedule. Job definitions are YAML files in `scheduled/`. The runner lives at `.opencode/cron/runner.ts`. To set up:

```bash
bun run .opencode/cron/runner.ts --install-defaults   # install jobs to ~/.openbench/cron/
bun run .opencode/cron/runner.ts --list                # list all jobs
bun run .opencode/cron/runner.ts --once morning-briefing  # run one immediately
```

Add to crontab (runs every minute, fires only when schedule matches):
```
* * * * * cd /path/to/openbench && bun run .opencode/cron/runner.ts
```

Default jobs: `morning-briefing` (weekdays 8am), `weekly-review` (Mondays 9am).

## Conventions

- The wiki (`wiki/`) uses PARA for human knowledge management. The memory system (`memory/`) is for agent session context. Agents write to `memory/`, humans write to `wiki/`.
- Skills live in `.opencode/skills/<category>/<name>/SKILL.md` and follow OpenCode's naming rules. Categories: `engineering/`, `agent/`, `research/`, `productivity/`, `ux/`, `design/`.
- Session summaries go in `memory/sessions/`, handoffs are optional.
- Artifacts are numbered sequentially (`01-`, `02-`, etc.) and must include a Table of Contents with anchor links to each section. Start from `artifacts/_TEMPLATE.md`.
- `wiki/0 Inbox/00 Scratchpad.md` is the human scratchpad — agents must never write to it.

## Token Efficiency (Headroom)

[Headroom](https://github.com/headroomlabs-ai/headroom) is a context optimization layer that compresses tool outputs, logs, files, and search results before they reach the model. 60–95% fewer tokens, same answers.

**How it works:** Content-type-aware compressors (JSON, code, logs, diffs, search results, text) feed into a Compress-Cache-Retrieve (CCR) store so compression stays reversible — the LLM can ask for the original whenever it wants.

**Initialization:**
```bash
pip install "headroom-ai[mcp]"              # one-time install
headroom mcp install                        # register with OpenCode
```

**Available MCP tools:**
- `headroom_compress(content)` — compress content on demand, returns compressed text + hash for retrieval
- `headroom_retrieve(hash, query?)` — retrieve original uncompressed content, optionally filtered by query
- `headroom_stats` — session compression statistics (tokens saved, cost saved, recent events)

**What NOT to use it for:**
- **Codebase search** — use Semble MCP (`semble_search`) for natural-language code queries
- **Session search** — use `python scripts/workspace/session-index.py search "<query>"`

**Layer model — each tool owns one question, cross-link by URL, never duplicate:**

| Question | Where | Granularity |
|----------|-------|-------------|
| Why does this matter? | `wiki/1 Projects/` top notes | Strategic / project |
| What's on the list? | Your tracker — initiatives → projects → issues | Issue / ticket |
| What did the agent decide? | `memory/sessions/` | Session |
| How does this work? | `artifacts/` | Document |

**When to create an issue:** A concrete piece of work with a definition of done — feature, bug, refactor, or task within a project. Not for research, strategy, or session notes.

**Cross-linking convention:** Issue descriptions link to relevant wiki notes and artifacts. Wiki top notes link to the appropriate tracker's initiative/project URLs. Session summaries reference issue IDs.

### Agent workflow with issues

**Canonical lifecycle** (single issue, human-gated):

```
"Work on ENG-5"
  → agent reads issue (description, comments, git branch name)
  → agent reads repo AGENTS.md + relevant code
  → agent implements on a branch, runs tests, commits
  → agent transitions issue: Backlog → In Progress
  → agent adds comment summarizing what was done + PR link
  → agent pauses — human reviews
  → on approval: human merges, agent transitions In Progress → Done
  → if changes needed: agent updates branch, re-comments
```

**Autonomy gates:**

- **Agent may** (no confirmation needed): read issues, transition Backlog → Todo → In Progress, add comments, attach PR links, create new issues from discovered work.
- **Agent may not** (requires explicit human ask): transition to Done (commits/merges are gated per workspace rules), deploy to staging/prod, delete issues, change priorities or labels without asking.
- **Agent should** (best practice): before starting work on an issue, check for blocking issues (`includeRelations: true`) and existing comments — don't start if context is missing or someone flagged a blocker.

**Backlog triage mode:**

When asked "what's next?" or "what should I work on?", the agent lists backlog issues, ranks by priority → initiative alignment → project fit, and pitches 1-2 options with a one-line rationale. Agent waits for human confirmation before starting implementation.
